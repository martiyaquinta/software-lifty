## 2026-06-30 — Open questions resolved via code investigation

### Q1: ¿`sub` == `users.id`? → NO
- Backend genera UUIDs con `defaultRandom()` (`schema/users.ts:4`).
- No hay sync entre Supabase y la tabla `users`. Nunca se llama a Supabase Admin API.
- **Decisión:** Agregar auto-create en authPlugin. Si el JWT es válido pero el usuario no existe en `users`, se inserta `{ id: sub, role: "driver" }`.

### Q2: ¿`role` de Supabase o DB? → DB
- El authPlugin (`middleware/auth.ts:30-38`) ya hace `SELECT role FROM users` y NO usa `payload.role`.
- **Decisión:** Sin cambios en este aspecto. El role se sigue resolviendo de la DB. Default: `"driver"`.

### Q3: ¿Conservar `/auth/refresh`? → NO
- El frontend lo usa como fallback (`client.ts:118-130`) cuando `supabase.auth.refreshSession()` falla.
- Post-migración se vuelve circular: el backend verifica con `SUPABASE_JWT_SECRET` pero `/auth/refresh` firmaría con el secreto viejo — sus tokens serían rechazados.
- **Decisión:** Borrar `/auth/refresh` completo (ruta, servicio, schema, `verifyRefresh`, `hashToken`, `buildAuthResponse`). Borrar el fallback del frontend.

### Q4: ¿change/reset password se usan? → NO
- El link "Olvidaste tu contraseña?" en `LoginCredentialsScreen.tsx:116` tiene `onPress={() => {}}` vacío.
- No hay screens de cambio/reset de password en el frontend.
- **Decisión:** Borrar `forgotPassword`, `resetPassword`, `forgotPasswordEmail`, `resetPasswordEmail`, `changePassword` (servicio + rutas + schemas).

## 2026-07-06 — Spec corrected after frontend audit

**Discovery:** The spec's core assumption was wrong — the frontend does NOT use Supabase Auth.
- Supabase Auth is fully disabled (`persistSession: false`, `autoRefreshToken: false`, `storage: undefined` in `lib/supabase.ts`).
- The frontend uses email/password auth against the backend exclusively (8 screens: Register, LoginCredentials, LoginOTP, ForgotPassword).
- Q3 and Q4 decisions above are now obsolete — `/auth/refresh`, `forgotPassword`, and `resetPassword` are all used by the frontend.

**Correction:** Replaced big-bang spec with 3-phase plan:
- Phase 1 (this cycle): Backend accepts JWTs from both sources. Auto-create users. Nothing deleted.
- Phase 2: Frontend migrates to Supabase Auth.
- Phase 3: Backend deletes internal auth dead code.

PR #29 (big-bang) was closed and reopened with Phase 1 only.

**Decisions for Phase 1:**
- `verifyAccess` tries `SUPABASE_JWT_SECRET` first, falls back to `JWT_SECRET`. `signAccess`/`signRefresh`/`hashToken` preserved.
- `authPlugin` auto-creates users with `password_hash = "supabase"` (placeholder for notNull column).
- Backend auth routes (`/register`, `/login`, `/refresh`, etc.) preserved — frontend still depends on them.
- All 206 tests pass, typecheck and lint clean.

## 2026-07-07 — Phase 2: Frontend migrates to Supabase Auth

**Scope:** mobile app only. Backend untouched (already accepts Supabase JWTs since Phase 1).

**Email verification decision:** OTP de 6 dígitos (email OTP), not magic links. Conserva las pantallas `OTPInput` existentes (Register + ForgotPassword) sin necesidad de deep-linking. Requiere "Email OTP" habilitado en el dashboard de Supabase.

**Changes:**
- `lib/supabase.ts`: habilitado Supabase Auth (`storage: AsyncStorage`, `persistSession: true`, `autoRefreshToken: true`). `AppState` wiring para `startAutoRefresh`/`stopAutoRefresh`. `onAuthStateChange` sincroniza la sesión con el authStore.
- `hooks/useAuth.ts`: reescrito sobre `supabase.auth.*`:
  - `useSignUp` → `signUp({ email, password })`
  - `useVerifyEmail` → `verifyOtp({ email, token, type: 'email' })`
  - `useResendCode` → `resend({ type: 'signup', email })`
  - `useForgotPassword` → `resetPasswordForEmail(email)`
  - `useResetPassword` → `verifyOtp({ type: 'recovery' })` + `updateUser({ password })`
  - `useLogin` → `signInWithPassword`
  - `useSignOut` → `POST /api/auth/logout` (best-effort) + `supabase.auth.signOut()`
  - `useRefreshToken` **eliminado** (lo maneja Supabase).
- `store/authStore.ts`: `setTokens(token, refreshToken)` → `setSession(token, userId?)`. Campo `refreshToken` eliminado (Supabase gestiona el refresh token internamente).
- `api/client.ts`: el interceptor 401 ahora refresca con `supabase.auth.refreshSession()` en vez de `POST /auth/refresh`. `/auth/refresh` sacado de `NO_REFRESH_PATHS`.
- `app/_layout.tsx` `SessionRestore`: hidrata desde `supabase.auth.getSession()` al arranque.
- `LoginOTPScreen`: corregido `{ token: otp }` → `{ code: otp }` (alineado con la firma del hook).

**Verification:** `bunx tsc --noEmit` limpio, `biome check` limpio en los archivos tocados. (Jest sigue roto por un problema de transform de RN preexistente — no relacionado a este cambio.)

**Nota operativa:** requiere en Supabase dashboard: Email provider habilitado, "Confirm email" con OTP, y templates de OTP para signup y recovery. El backend sigue exponiendo `/auth/*` hasta Phase 3.
