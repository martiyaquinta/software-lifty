## 2026-06-30 — Open questions resolved via code investigation

### Q1: ¿`sub` == `users.id`? → NO
- Backend genera UUIDs con `defaultRandom()` (`schema/users.ts:4`).
- No hay sync entre Supabase y la tabla `users`. Nunca se llama a Supabase Admin API.
- **Decisión:** Agregar auto-create en authPlugin. Si el JWT es válido pero el usuario no existe en `users`, se inserta `{ id: sub, role: "driver" }`.

### Q2: ¿`role` de Supabase o DB? → DB
- El authPlugin (`middleware/auth.ts:30-38`) ya hace `SELECT role FROM users` y NO usa `payload.role`.
- **Decisión:** Sin cambios en este aspecto. El role se sigue resolviendo de la DB. Default: `"driver"`.

### Q3: ¿Conservar `/auth/refresh`? → NO
- El frontend lo usa como fallback (`client.ts:118-130`) cuando `supabase.auth.refreshSession()` falla.
- Post-migración se vuelve circular: el backend verifica con `SUPABASE_JWT_SECRET` pero `/auth/refresh` firmaría con el secreto viejo — sus tokens serían rechazados.
- **Decisión:** Borrar `/auth/refresh` completo (ruta, servicio, schema, `verifyRefresh`, `hashToken`, `buildAuthResponse`). Borrar el fallback del frontend.

### Q4: ¿change/reset password se usan? → NO
- El link "Olvidaste tu contraseña?" en `LoginCredentialsScreen.tsx:116` tiene `onPress={() => {}}` vacío.
- No hay screens de cambio/reset de password en el frontend.
- **Decisión:** Borrar `forgotPassword`, `resetPassword`, `forgotPasswordEmail`, `resetPasswordEmail`, `changePassword` (servicio + rutas + schemas).
